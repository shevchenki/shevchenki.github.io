Deploy EKS cluster

I. Preparation​

1. Notes:​

  • Please switch [aws account id] with your account id.

  • Overall configuration diagram:

    • Network configuration

    • Cluster configuration

  • Source code: template-terraform

  • Deployment scope:

    • EKS Cluster.
    • EKS Worker Node: AutoScaler configuration.
    • AWS Application LoadBalancer for Ingress.
    • ECR

2. Environment​

  • Amazon Linux
  • AWS CLI Version: 2.0.8
  • Install eksctl on Linux
    curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
    sudo mv /tmp/eksctl /usr/local/bin
    eksctl version

II. Create EKS Cluster​

1. Create EKS Cluster with Terraform​

  • Create resources based on template-terraform

    • Run init
      terraform init
    • Run plan
      terraform plan
    • Run apply
      terraform apply

  • Confirmation of results:

    aws eks describe-cluster --name sandbox_eks_cluster --query cluster.status --profile sandbox-env
    > "ACTIVE"

2. Setting kubectl config​

  • Setting kubectl config:

    aws eks update-kubeconfig --kubeconfig ~/.kube/config-eks --name sandbox_eks_cluster --profile sandbox-env
    export KUBECONFIG=~/.kube/config-eks

  • Confirm operation (Get service information):

    kubectl get svc
    >
    > NAME         TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
    > kubernetes   ClusterIP   172.20.0.1   <none>        443/TCP   14m

  • Get node information:

    kubectl get nodes
    > NAME                                           STATUS   ROLES    AGE   VERSION
    > ip-10-0-3-61.ap-northeast-1.compute.internal   Ready    <none>   10m   v1.24.11-eks-a59e1f0

III. Deployment of Cluster Autoscaler​

1. Description​

  • Cluster Autoscaler is a tool that automatically adjusts the size of a Kubernetes cluster when any of the following conditions are met.
    • There are pods in the cluster that have failed to run due to lack of resources.
    • There are nodes in the cluster that have not been used for a long time and those pods can be placed on other existing nodes.
  • AWS Cluster Autoscaler extends worker nodes in any given Auto Scaling group and runs as a deployment within the cluster.

2. Download sample deployment files​

wget https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

cluster-autoscaler

3. Change EKS cluster name (eks-cluster) and environment variables in cluster-autoscaler-autodiscover.yaml​

    ...          
        command:
        - ./cluster-autoscaler
        - --v=4
        - --stderrthreshold=info
        - --cloud-provider=aws
        - --skip-nodes-with-local-storage=false
        - --expander=least-waste
        - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/[eks-cluster-name]
    ...

4. Deploy Cluster Autoscaler​

kubectl apply -f cluster-autoscaler-autodiscover.yaml

5. Check the Cluster Autoscaler deployment logs if deployment errors​

kubectl logs -f deployment/cluster-autoscaler -n kube-system

IV. Deploy Application Load Balancer Ingress​

1. Describe IAM OIDC provider that integrated to eks cluster​

aws eks describe-cluster --name sandbox_eks_cluster --query "cluster.identity.oidc.issuer" --output text --profile sandbox-env
  • Note the oidc output "oidc.eks.[aws region].amazonaws.com/id/F376C***"

    https://oidc.eks.ap-northeast-1.amazonaws.com/id/8B5F0****

2. Install Add-ons​

vpc-cni​

  • Download vpc-cni-trust-policy.json and create AmazonEKSVPCCNIRole

    aws iam create-role \
    --role-name AmazonEKSVPCCNIRole \
    --assume-role-policy-document file://"vpc-cni-trust-policy.json" \
    --profile sandbox-env

  • Attach AmazonEKS_CNI_Policy to AmazonEKSVPCCNIRole

    aws iam attach-role-policy \
    --policy-arn arn:aws:iam::aws:policy/AmazonEKS_CNI_Policy \
    --role-name AmazonEKSVPCCNIRole \
    --profile sandbox-env

  • Run the following command to annotate the aws-node service account with the ARN of the IAM role that you created previously.

    kubectl annotate serviceaccount \
        -n kube-system aws-node \
        eks.amazonaws.com/role-arn=arn:aws:iam::[aws account id]:role/AmazonEKSVPCCNIRole
    • serviceaccount/aws-node annotated -> go to next

  • See which version of the add-on is installed on your cluster.

    kubectl describe daemonset aws-node --namespace kube-system | grep amazon-k8s-cni: | cut -d : -f 3
    > v1.12.6-eksbuild.2

  • See which type of the add-on is installed on your cluster. Depending on the tool that you created your cluster with, you might not currently have the Amazon EKS add-on type installed on your cluster. 

    aws eks describe-addon --cluster-name sandbox_eks_cluster --addon-name vpc-cni --query addon.addonVersion --output text --profile sandbox-env
    • An error occurred (ResourceNotFoundException) when calling the DescribeAddon operation: No addon: vpc-cni found in cluster: sandbox_eks_cluster -> go to next

  • Create the add-on using the AWS CLI.

    aws eks create-addon --cluster-name sandbox_eks_cluster --addon-name vpc-cni --addon-version v1.12.6-eksbuild.2 \
        --service-account-role-arn arn:aws:iam::[aws account id]:role/AmazonEKSVPCCNIRole --profile sandbox-env

  • Update to latest version

    aws eks update-addon --cluster-name sandbox_eks_cluster --addon-name vpc-cni --addon-version  v1.13.0-eksbuild.1 \
        --service-account-role-arn arn:aws:iam::[aws account id]:role/AmazonEKSVPCCNIRole \
        --resolve-conflicts PRESERVE --profile sandbox-env

coredns add-on​

  • See which version of the add-on is installed on your cluster.

    aws eks describe-addon --cluster-name sandbox_eks_cluster --addon-name coredns --query addon.addonVersion --output text --profile sandbox-env
    • An error occurred (ResourceNotFoundException) when calling the DescribeAddon operation: No addon: coredns found in cluster: sandbox_eks_cluster

  • Create the add-on using the AWS CLI.

    aws eks create-addon --cluster-name sandbox_eks_cluster --addon-name coredns --addon-version v1.10.1-eksbuild.1 --profile sandbox-env

kube-proxy​

  • Create the add-on using the AWS CLI.
    aws eks create-addon --cluster-name sandbox_eks_cluster --addon-name kube-proxy --addon-version v1.27.1-eksbuild.1 --profile sandbox-env

3. Create ALBIngressControllerIAMPolicy​

  • Refer: https://docs.aws.amazon.com/ja_jp/eks/latest/userguide/aws-load-balancer-controller.html
  • alb-ingress-controller
curl -O https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.4.7/docs/install/iam_policy.json

aws iam create-policy \
    --policy-name ALBIngressControllerIAMPolicy \
    --policy-document file://iam_policy.json \
    --profile sandbox-env
  • Note policy-arn "arn:aws:iam::[aws account id]:policy/ALBIngressControllerIAMPolicy".

    arn:aws:iam::[aws account id]:policy/ALBIngressControllerIAMPolicy

4. Create IAM ROLE for ALB Ingress Controller​

4.1 Create assume-role-policy-document.json​

  • Switch between "Federated" and "StringEquals" in the above oidc output
    {
    "Version": "2012-10-17",
    "Statement": [
        {
        "Effect": "Allow",
        "Principal": {
            "Federated": "arn:aws:iam::[aws account id]:oidc-provider/oidc.eks.ap-northeast-1.amazonaws.com/id/8B5F0********"
        },
        "Action": "sts:AssumeRoleWithWebIdentity",
        "Condition": {
            "StringEquals": {
            "oidc.eks.ap-northeast-1.amazonaws.com/id/8B5F0********:aud": "sts.amazonaws.com",
            "oidc.eks.region-code.amazonaws.com/id/8B5F0********:sub": "system:serviceaccount:kube-system:aws-load-balancer-controller"
            }
        }
        }
    ]
    }

4.2 Create Role by AWS CLI​

aws iam create-role \
    --role-name eks-alb-ingress-controller \
    --assume-role-policy-document file://assume-role-policy-document.json \
    --profile sandbox-env

4.3 Applying role policy (using policy-arn above)​

aws iam attach-role-policy --policy-arn arn:aws:iam::[aws account id]:policy/ALBIngressControllerIAMPolicy --role-name eks-alb-ingress-controller --profile sandbox-env

4. Deploy ALB Ingress Controller.​

Install cert-manager using one of the following methods to inject certificate configuration into the webhooks.

If your nodes have access to the quay.io container registry, install cert-manager to inject certificate configuration into the webhooks.

```shell
kubectl apply \
    --validate=false \
    -f https://github.com/jetstack/cert-manager/releases/download/v1.5.4/cert-manager.yaml
```

  • Download aws-load-balancer-controller manifest

    curl -Lo v2_4_7_full.yaml https://github.com/kubernetes-sigs/aws-load-balancer-controller/releases/download/v2.4.7/v2_4_7_full.yaml

  • Add information to v2_4_7_full.yaml (vpc, region, eks name)

    spec:
      containers:
      - args:
        - --ingress-class=alb
        - --cluster-name=sandbox_eks_cluster
        - --aws-vpc-id=[vpc id]
        - --aws-region=[aws region]

  • Add eks.amazonaws.com/role-arn annotations to ServiceAccount

    ---
    apiVersion: v1
    kind: ServiceAccount
    metadata:
    labels:
        app.kubernetes.io/component: controller
        app.kubernetes.io/name: aws-load-balancer-controller
    name: aws-load-balancer-controller
    namespace: kube-system
    annotations:
        eks.amazonaws.com/role-arn: arn:aws:iam::[aws account id]:role/eks-alb-ingress-controller

  • Deploy ALB Ingress Controller

    kubectl apply -f v2_4_7_full.yaml

  • Deploy IngressClass and IngressClassParams:

    curl -Lo v2_4_7_ingclass.yaml https://github.com/kubernetes-sigs/aws-load-balancer-controller/releases/download/v2.4.7/v2_4_7_ingclass.yaml
    k apply -f v2_4_7_ingclass.yaml

  • Verify that the controller is installed.

    kubectl get deployment -n kube-system aws-load-balancer-controller
